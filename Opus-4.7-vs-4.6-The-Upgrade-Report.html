I. Executive Summary

Three things you need to know

1. Opus 4.7 wins 12 of 14 benchmarks, with the largest gains in tool orchestration (+15.5pp), visual acuity (+44.0pp), and coding (+6.8pp on SWE-bench Verified). [1][7] HIGH

2. The per-token price is unchanged ($5/$25 per MTok) [3], but a new tokeniser inflates token counts by up to 35% [4][13], making real-world costs higher despite the identical rate card. HIGH

3. Our own experiments show all three Claude tiers achieve 100% instruction compliance on structured prompts, but no model can accurately count its own words, a novel finding from 12 first-hand tests. PRIMARY RESEARCH

Plain English

Opus 4.7 is smarter than Opus 4.6 at almost everything, especially coding, tool use, and reading charts. The price per word hasn’t changed, but Opus 4.7 uses more words to say the same thing, so your monthly bill could go up 15-35%. We ran 12 experiments ourselves to verify what the benchmarks claim.

IV. Five Key Findings

Finding 1
The best coding model just got 8.4% better HIGH

SWE-bench Verified jumped from 80.8% (Opus 4.6) to 87.6% (Opus 4.7) [1][7]. That is the single largest version-to-version gain on this benchmark in the Claude family. Opus 4.7 is now the top-performing generally available model on real GitHub issues. [11]

Finding 2
Tool use improved by 25%. This is the agentic upgrade. HIGH

MCP-Atlas leapt from 61.8% (Opus 4.6) to 77.3% (Opus 4.7) [1][7], a 15.5 percentage-point gain. If you build agents that call multiple tools, Opus 4.7 is a step change over Opus 4.6, not an increment.

Finding 3
Same rate card. Up to 35% more tokens per task. HIGH

Opus 4.7 uses a new tokeniser that converts the same text into 1.0-1.35× more tokens than Opus 4.6. [4][13][15] The sticker price didn’t change between Opus 4.6 and Opus 4.7. The meter runs faster.

Finding 4
Vision went from adequate to extraordinary. HIGH

Visual acuity jumped from 54.5% (Opus 4.6) to 98.5% (Opus 4.7). [1][4][9] Image resolution increased to 2,576px / 3.75 megapixels on Opus 4.7, more than 3× Opus 4.6’s limit. [4]

Finding 5
No model can count its own words. PRIMARY RESEARCH

We asked all three Claude models to write exactly 50 words. Opus produced 44, Sonnet 51, Haiku 62. All three claimed they hit 50 exactly. This is a genuine limitation of current LLMs that no benchmark captures.

The silver lining

Hex’s early-access testing found that low-effort Opus 4.7 matches medium-effort Opus 4.6 on quality. When upgrading from Opus 4.6 to Opus 4.7, the trick is using the right effort level, not the highest one. MED

Enterprise maths (illustrative estimate)

A hypothetical enterprise running 100,000 daily Opus queries at 500 output tokens each:

  • Daily cost (standard): ~$1,250 / day = ~$456,000 / year [3]
  • With Batch API: ~$228,000 / year
  • The Opus 4.7 tokeniser bump (vs Opus 4.6) could add an estimated $0-$160,000 / year on the input side alone. [15]

These are simplified estimates for illustration only. Actual costs vary significantly based on prompt length, caching, effort level, and the ratio of input to output tokens. Run your own cost modelling before budgeting.

VIII. Primary Research: The Prompt Experiments

“Don’t tell me what the model can do. Show me what it does.”

This section presents the results of 12 first-hand experiments conducted on 18 April 2026 using Claude Code’s Task tool. Three models were directly tested (Opus 4.6, Sonnet 4.5, and Haiku 4.5) across two experiment batteries. Each sub-agent self-reported its model ID, confirming correct routing. All results were manually verified by the research team. PRIMARY RESEARCH

Note on Opus 4.7 vs Opus 4.6: At time of testing, the Opus API endpoint serves Opus 4.6 (claude-opus-4-6). Opus 4.7 is not yet available for direct API testing. Where Opus 4.7 appears in the scorecard below, results are projected from published benchmark data [1][4][7], not first-hand tested. All first-hand experiment results in this section are from Opus 4.6. The full prompts used in each experiment are shown below so readers can reproduce these tests once Opus 4.7 becomes available.
Methodology note. Each experiment was run once per model (n=1). LLM outputs are non-deterministic: the same prompt may produce different results on subsequent runs. These experiments are qualitative observations, not controlled studies. They illustrate typical model behaviour but should not be treated as statistically significant benchmarks. No temperature or sampling parameters were set (defaults were used). We publish the exact prompts above so that readers can reproduce, extend, or challenge these results independently.

Why This Matters More Than Benchmarks

Published benchmarks test models under controlled conditions with known answers. Our experiments test real-world behaviours: Can the model follow instructions? Can it count? Does it produce better code than the cheaper alternative? These are the questions that determine your spend, not a leaderboard number.

XI. New Features & Breaking Changes

What’s new in Opus 4.7 (vs Opus 4.6) [4]

  • xhigh effort level: ~80-90% of max quality at 40-50% latency. [5]
  • Adaptive thinking only: Manual budget_tokens removed (was available in Opus 4.6). [6][14]
  • Higher-resolution images: 2,576px / 3.75MP (was 1,568px / 1.15MP on Opus 4.6). [4]
  • Stricter instruction following: Low-effort Opus 4.7 is genuinely brief. [4]
  • More direct tone: Less validation-forward phrasing, fewer emoji. [4]

XII. Who Should Upgrade

The guidance below is based on published benchmarks and our own observations. It is not a substitute for testing on your own workloads. Model behaviour can vary significantly across domains, languages, and prompt styles.

Upgrade to Opus 4.7 now

  • Coding agents: Opus 4.7 gains +6.8pp on SWE-bench Verified and +12.0pp on CursorBench vs Opus 4.6. [1]
  • Multi-tool agents: Opus 4.7 gains +15.5pp on MCP-Atlas vs Opus 4.6. [1]
  • Vision workflows: Opus 4.7 achieves 98.5% visual acuity (was 54.5% on Opus 4.6). [1][4]
  • Anyone using “max” on Opus 4.6: Switch to “xhigh” on Opus 4.7. [5]

Wait and test first

  • Web browsing agents: Opus 4.7 regressed 4.7pp on BrowseComp vs Opus 4.6. [1]
  • Security tooling: Opus 4.7 shows a small Cybersec regression (−0.7pp vs Opus 4.6). [1][12]
  • Cost-sensitive pipelines: Opus 4.7’s tokeniser can silently inflate bills vs Opus 4.6. [13][15]
  • Code using temperature/top_p: Returns 400 errors on Opus 4.7. [4]

Stay on Opus 4.6

  • If your pipeline depends on budget_tokens and you need time to refactor (removed in Opus 4.7).
  • If your unit economics are tight and you can’t absorb the potential 15-35% cost increase from Opus 4.7’s new tokeniser.

XIII. Glossary

Plain-English definitions of every benchmark and technical term used in this report.

SWE-bench Verified: Can the model fix real GitHub bugs? A curated set of 500 verified issues from popular open-source repositories. The model receives the issue description and must produce a working patch.

SWE-bench Pro: A harder subset of SWE-bench with more complex, multi-file bugs that require deeper understanding of codebases.

SWE-bench Multilingual: The same test methodology as SWE-bench, but applied to non-English codebases (Java, C++, Go, etc.) to measure cross-language capability.

GPQA Diamond: PhD-level science questions spanning physics, chemistry, and biology. Questions are written and verified by domain experts to be genuinely challenging.

MCP-Atlas: Can the model orchestrate multiple tools in sequence? Measures the ability to plan and execute multi-step tool-use chains via the Model Context Protocol.

CursorBench: Real IDE coding agent tasks that simulate how developers use AI-powered editors like Cursor. Measures code generation, editing, and navigation.

Terminal-Bench 2.0: CLI and terminal agent tasks: navigating filesystems, running commands, parsing output, and solving problems entirely through the command line.

OSWorld: Computer use tasks requiring the model to interact with graphical UIs: clicking buttons, typing in fields, navigating between applications.

CharXiv: Reading and interpreting scientific charts from academic papers. Tests whether the model can extract data, identify trends, and answer questions about figures.

Visual Acuity: How precisely can the model read fine detail in images? Measures the ability to resolve small text, subtle colours, and intricate patterns.

BrowseComp: Web browsing and information retrieval accuracy. The model must navigate web pages, find specific information, and synthesise answers.

Finance Agent: Researching SEC filings and financial documents. The model must locate, read, and reason about regulatory disclosures and financial data.

Cybersec Vuln Repro: Reproducing known software security vulnerabilities. The model receives a CVE description and must demonstrate the vulnerability in a controlled environment.

Tokeniser: The algorithm that splits text into tokens before the model processes it. Different tokenisers can produce different token counts for the same text, directly affecting cost.

XIV. Limitations

Transparency about what this report does and does not prove:

  • Sample size: n=1 per model per task. Each experiment was run once. LLM outputs are non-deterministic, and a single run cannot establish statistical significance. Results may differ on repeat runs.
  • No API-level token counting. Opus 4.7 tokeniser inflation figures (1.0-1.35x vs Opus 4.6) are from published reports [4][13], not our own measurement. We did not count tokens consumed by our experiments.
  • Sub-agent experiments measure output characteristics, not latency or cost. We captured word counts, character counts, and instruction compliance, not response time, throughput, or dollar cost per query.
  • Regression data from published benchmarks only. BrowseComp and Cybersec regression figures come from Anthropic’s published results [1], not from our own testing. We cannot independently verify these regressions.
  • No head-to-head quality grading. We measured quantity and constraint compliance, not subjective or systematic quality scoring (e.g. human preference rankings, ELO, or blind evaluations).
  • Experiments used Opus 4.6, not Opus 4.7. At time of testing, the Opus model available via Claude Code was Opus 4.6 (claude-opus-4-6). All Opus 4.7 claims in this report are from published benchmarks, not first-hand experiments. Direct Opus 4.7 vs Opus 4.6 comparisons in the primary research section are therefore limited to projected benchmark data.
  • No multi-language or multi-domain testing. All prompts were in English. Performance on non-English tasks or specialised domains (medical, legal, etc.) was not tested.
  • Cost estimates are simplified. Real-world API costs depend on prompt length, system prompt caching, effort level selection, batch vs. real-time, and input/output token ratios. Our cost tables use output-only pricing for illustration.
  • Benchmarks are vendor-reported. Most benchmark figures originate from Anthropic’s own publications. Independent third-party verification of all claims was not possible at time of writing.
  • Snapshot in time. This report covers publicly available information as of April 2026. Model capabilities, pricing, and API behaviour may change without notice after publication.
